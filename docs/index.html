PI For Principal Investigators ▸

A non-operational overview: what this pipeline measures, how to read the results, and why they are reproducible.

What it does, scientifically

The pipeline profiles the bacterial/archaeal community in each sample by sequencing the 16S rRNA gene (V4 region, 515F/806R primers). It uses the Earth Microbiome Project protocol and QIIME 2. Raw sequencing reads are denoised into amplicon sequence variants (ASVs) — exact sequences, finer resolution than traditional 97% OTUs — using DADA2, then classified against the Greengenes reference and summarized with standard diversity metrics.

What each result answers

ResultQuestion it answersWhere
taxa-bar-plots.qzvWho is present, and at what relative abundance, per sample/group?Deliverable
Alpha diversityHow rich/even is each sample? (Shannon, Faith PD, evenness, observed features)Deliverable
Alpha-group significanceDoes within-sample diversity differ between metadata groups?core-metrics-results/
Beta diversity + PCoA/EmperorHow different are whole communities between samples/groups? (Bray-Curtis, Jaccard, UniFrac)core-metrics-results/
Beta-group significance (PERMANOVA)Are between-group community differences statistically significant?core-metrics-results/

What you receive

The shared bundle (deliverable/ and a .tar.gz) contains the interpretable results and the underlying data artifacts, so collaborators can do their own downstream analysis: the ASV table, representative sequences, rooted tree, taxonomy, taxa bar plots, diversity results, denoising/QC summaries, the sample metadata, and a checksummed MANIFEST.txt. Intermediate working files are left out to keep the bundle small.

Reproducibility & provenance

  • Every result file carries its own provenance. QIIME 2 .qza/.qzv files embed the complete chain of actions, parameters, and software versions used to produce them. Open any .qzv at view.qiime2.org → Provenance tab.
  • Citations are embedded too (citations.bib inside each artifact) — the correct references for QIIME 2, DADA2, MAFFT/FastTree, UniFrac, and the Greengenes classifier for a methods section.
  • Bundle integrity is recorded in MANIFEST.txt (md5 checksum + size per file, plus run parameters and QIIME version).

Key methods to cite

QIIME 2 (Bolyen et al. 2019); DADA2 (Callahan et al. 2016); Greengenes 13-8 reference; MAFFT + FastTree for phylogeny; UniFrac for phylogenetic beta diversity. Exact versions are in each artifact's provenance.

User User Guide — how to run it ▸

How to run the pipeline end to end.

Prerequisites

  • A QIIME 2 environment on PATH (conda activate scope-env or a QIIME 2 amplicon install). The native qiime, plus wget and biom, must be callable.
  • Internet access for setup (downloads the classifier).
  • Roomy scratch space — DADA2/demux write large temp files (see --tmpdir).

1. Prepare inputs

  • Raw reads — Illumina bcl2fastq "Undetermined" output, three files whose names contain _R1_, _R2_, and _I1_.
  • A mapping file — tab-delimited QIIME metadata whose name contains mapping. It must have a #Sample ID column and a barcode column named BarcodeSequence (12 bp Golay barcodes for EMP).

2. Stage the run (setup)

python scripts/qiime-console.py setup /path/to/raw_source /path/to/base_dir

This creates base_dir/input/{raw_data,mapping.txt}, renames the raw files to forward/reverse/barcodes.fastq.gz, and downloads the classifier into input/.

3. Run the workflow

python scripts/qiime-console.py run workflow /path/to/base_dir \
    --p-trunc-len-f 250 --p-trunc-len-r 250 \
    --p-sampling-depth 20000 \
    --beta-diversity-group-by Sample_Plate
FlagDefaultMeaning
--p-trunc-len-f / -r0Truncate reads at this position (0 = none). Set from the demux quality plots.
--p-sampling-depth20000⚠ Footgun: every sample with fewer reads is dropped from diversity. Pick it from the summaries — don't accept the default blindly.
--p-max-depth10000Max depth for rarefaction curves.
--p-steps10Number of rarefaction steps.
--beta-diversity-group-bynoneMetadata column(s) for beta-group significance. Repeatable; validated up front.
--barcode-column-nameBarcodeSequenceMapping column holding the barcodes.
--p-n-threads0DADA2 threads (0 = all cores).
--tmpdirautoScratch dir for QIIME temp files; avoids small RAM-backed /tmp.

4. Check quality

  • output/demux/demux-full.qzv — per-sample read counts and quality; use it to choose truncation lengths and a sensible sampling depth.
  • output/stats-dada2.qzv — how many reads survived filtering, merging, and chimera removal per sample.
  • Open any .qzv by dragging it onto view.qiime2.org.

5. Package for sharing (automatic, or on demand)

run workflow already builds the deliverable. To (re)build it standalone:

python scripts/qiime-console.py package /path/to/base_dir --format both \
    --p-trunc-len-f 250 --p-trunc-len-r 250 --p-sampling-depth 20000 \
    --beta-diversity-group-by Sample_Plate

--format is folder, tar, or both (default). Any parameter flags you pass are recorded in MANIFEST.txt.

Running individual steps (advanced)

python scripts/qiime-console.py run demux   /path/to/base_dir   # import + demux + summarize
python scripts/qiime-console.py run denoise /path/to/base_dir   # dada2 denoise-paired

Normally you just run run workflow, which chains everything.

Dev Developer Guide — how it's built ▸

Code architecture of the QIIME 2 pipeline and how to extend it.

Entry point

Everything runs through scripts/qiime-console.py, an argparse CLI. bin/qiime → scripts/qiime-consol.py → qiime-console.py (two symlinks; note the truncated middle name).

CommandFunctionPurpose
setup <src> <base>stage_dataDiscover/rename raw FASTQ, copy mapping, fetch classifier
run workflow <base>run_workflowFull pipeline (chains all stages + auto-package)
run demux <base>demux_dataStandalone import + demux + summarize
run denoise <base>denoise_dataStandalone DADA2 denoise-paired
package <base>package_resultsBuild the shareable deliverable + manifest
tool <name> <args>run_toolThin passthrough to qiime

The input/output + link_output pattern

Steps write to output/ and read from input/. link_output(src, input_dir) symlinks each new output/ artifact back into input/ by basename, so the next step finds it there. It uses os.path.lexists to detect and replace stale or dangling symlinks from prior runs.

Stage functions (call order)

  1. import → demux → dada2 denoise → feature-table summarize / tabulate-seqs / stats tabulate.
  2. run_phylogeny_analysis — despite the name, runs the tree and a second tabulate-seqs (rep-seqs.qzv), classify-sklearn, taxonomy tabulate, and the taxa barplot. Taxonomy runs before diversity.
  3. run_diversity_analysis — core-metrics-phylogenetic, three alpha-group-significance calls, alpha-rarefaction, beta-group-significance (one per column).
  4. run_relative_abundance_of_taxonomy — taxa collapse levels 1–7, relative-frequency, export, biom convert.
  5. package_results (+ write_manifest) — best-effort, wrapped so a packaging failure never fails an otherwise-complete run.
Gotcha — alpha-rarefaction is coded twice, runs once. Both run_phylogeny_analysis and run_diversity_analysis call it with the same output dir. Phylogeny runs first, so the diversity call hits the skip-if-exists guard. The other call is dead code, not a second run.

Packaging internals

  • DELIVERABLE_FILES / DELIVERABLE_DIRS define the curated export set; entries may be a list of alternative paths (first existing wins) to tolerate layout drift.
  • DELIVERABLE_DIRS are copied with ignore=_ignore_except(DELIVERABLE_DIR_EXTS) so only .qza+.qzv are shipped — redundant .tsv/.biom/.html/logs are excluded.
  • package_results clears deliverable/ before rebuilding, validates --format, and for --format tar removes the staging folder.
  • write_manifest checksums every bundled file (md5), records run params + QIIME version, and is written before the tarball so it's archived.

Other internals

  • resolve_tmpdir — picks a roomy, disk-backed temp dir and exports TMPDIR/TMP/TEMP so native qiime inherits it. Warns below MIN_TMP_FREE_GB = 50.
  • Metadata columns — read_metadata_columns strips per-field whitespace and a leading #; resolve_metadata_column does exact-then-fuzzy matching so typos fail fast.

How to add a step

  1. Write the artifact to output/ with a skip-if-exists guard.
  2. link_output(new_artifact, input_dir) (guard the link if the step can fail).
  3. If it should be shared, add it to DELIVERABLE_FILES/DELIVERABLE_DIRS.
  4. Add a row to both tables in PIPELINE.md and re-run scripts/check_docs_tables.py.

Two-generation split

  • QIIME 2 (current): qiime-console.py (authoritative), the shell scripts, commands.txt, and an in-progress CWL port.
  • QIIME 1 (legacy): the OTU-based CWL workflows under CWL/, wrapping QIIME 1.9.1 scripts under mgrast/qiime:1.0.

Known issues & recent fixes

  • Fixed: run demux works; --barcode-column-name honored; metadata headers trimmed; shell classifier name corrected gg-18-8 → gg-13-8.
  • Tracked: remaining metadata-hardening items (issue #5).
Maint Maintainer Guide — how to operate it ▸

Operational and release concerns.

Runtime environment

  • QIIME 2 must be on PATH (conda activate scope-env, or a QIIME 2 amplicon distribution). The pipeline shells out to native qiime with no path resolution.
  • Also required on PATH: wget (classifier download) and biom (rel-abundance TSV export).
  • Python 3 for qiime-console.py (bytecode is 3.12).
  • Scratch space: DADA2/demux are I/O heavy — see resolve_tmpdir and MIN_TMP_FREE_GB = 50.

Classifier provisioning

setup fetches the Greengenes 13-8 515F/806R naive-Bayes classifier via wget:

https://data.qiime2.org/classifiers/sklearn-1.4.2/greengenes/
gg-13-8-99-515-806-nb-classifier.qza

The URL is pinned to sklearn 1.4.2 — it must match the scikit-learn version in the QIIME 2 environment, or classify-sklearn will error. When bumping QIIME 2, update this URL to the matching sklearn-<ver> path. No .qza classifiers are committed to the repo.

Repo housekeeping

  • Submodule: ebi-metagenomics-cwl is declared but not checked out. Init only to regenerate CWL wrappers.
  • Tests: only CWL job-input YAMLs under tests/jobs/ — no test runner, referenced fixtures not committed.
  • Docs drift guard: scripts/check_docs_tables.py verifies the PIPELINE.md tables still match the code; run it in CI or before tagging.

Release / versioning

  • Docker images follow major.minor.build where build is a timestamp.
  • No formal tag/release for the Python pipeline; changes land via PRs to master. Per-run QIIME version is captured in MANIFEST.txt.

Roadmap

  • QIIME 2 CWL port in progress: CWL/Tools/qiime2/import.cwl and scripts/wf.1.0.cwl are the seed; wf.1.2.cwl is an empty stub.
  • Consolidate the legacy QIIME 1 CWL and shell scripts as the Python driver becomes the single source of truth.

Pipeline flow

Nine stages, in real execution order. Each step reads from input/ and writes to output/; artifacts are symlinked back into input/ for the next step.

flowchart TD
    setup([setup: stage FASTQ + mapping + fetch classifier])
    imp[import: EMPPairedEndSequences]
    demux[demux emp-paired + summarize]
    dada2[dada2 denoise-paired: table, rep-seqs, stats]
    tree[phylogeny: align-to-tree-mafft-fasttree, rooted tree]
    tax[classify-sklearn + taxonomy + taxa barplot]
    core[diversity core-metrics-phylogenetic]
    sig[alpha + beta group-significance, rarefaction]
    rel[taxa collapse L1-7 + relative-frequency]
    pkg([package: deliverable + tar.gz + MANIFEST])
    setup --> imp --> demux --> dada2 --> tree --> tax --> core --> sig --> rel --> pkg
!
Two things surprise readers: taxonomy runs before diversity (inside the misleadingly-named run_phylogeny_analysis), and alpha-rarefaction is coded in two functions but runs once (the second call hits a skip-if-exists guard).

What we deliberately ship — and don't

QIIME emits the same result in several forms. The rule is vizzes (.qzv) + data artifacts (.qza), nothing else. Redundant forms stay out of the bundle.

viz/demux-full/ (HTML) → demux-full.qzv
The .qzv is the HTML, zipped with provenance — view it at view.qiime2.org.
rel-phyla-table.*.tsv / biom → .qza feature tables
Exported TSV/biom duplicate data already inside the .qza; regenerate on demand.
rep-seqs.qzv → rep-seqs-dada2.qzv
The same visualization is produced twice; the bundle ships one.
non-.qza/.qzv inside result dirs → .qza + .qzv only
Result directories are filtered on copy, so any stray .tsv/.html/log QIIME drops there is not shipped.
⇄
Why a file appears in two places. After writing an artifact, the driver symlinks it by name into input/ for the next step. So a UniFrac significance .qzv lives once in core-metrics-results/ with a symlink in input/ — one real file, and the symlink is not part of the deliverable.

Provenance & reproducibility

Every .qza/.qzv embeds QIIME 2's full provenance graph — the exact actions, parameters, plugin and framework versions, and a citations.bib. Drag any .qzv onto view.qiime2.org and read the Provenance tab.

At the bundle level, MANIFEST.txt records an md5 checksum and byte size for every shipped file, plus the run parameters and QIIME version — so a shared deliverable is both self-describing and verifiable on transfer.
